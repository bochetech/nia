NIA Widget

Chat de IA embebible en cualquier sitio web. Un solo <script> tag. Aislamiento total via Shadow DOM. Theming por tenant.

⚛ Preact 10 🎭 Shadow DOM 🔒 Sin conflictos CSS 📦 IIFE · sin dependencias ♿ WCAG 2.1 AA

¿Qué es el widget?

El NIA Widget es un chat de IA embebible que los tenants insertan en sus sitios web con un único <script> tag. Se conecta al backend de NIA para procesar mensajes, gestionar el estado de la conversación (FSM) y capturar leads.

Está construido con Preact (3 KB, API idéntica a React) y aislado completamente del CSS del sitio host mediante Shadow DOM. El theming (colores, logo, textos) se configura por tenant desde el backend.

✅
El widget no tiene dependencias externas en runtime. El bundle IIFE incluye todo lo necesario (Preact + estilos + lógica).

Quickstart

Dos formas de insertar el widget. Elige la que prefieras:

Opción A — window.NIA_CONFIG RECOMENDADA

Inyecta la configuración antes del script. Funciona en cualquier posición del DOM, incluyendo al final del <body>.

<!-- 1. Config inline (siempre antes del widget script) -->
<script>
  window.NIA_CONFIG = {
    tenant:           "mi-empresa",
    apiUrl:           "https://api.nia.io",
    tenantManagerUrl:  "https://api.nia.io",
    position:         "bottom-right"   // u "bottom-left"
  };
</script>
<!-- 2. Bundle del widget -->
<script src="https://cdn.nia.io/widget/nia-widget.iife.js"></script>

Opción B — atributos data-*

Funciona cuando el script se coloca en el <head> (donde document.currentScript aún es válido).

<head>
  <script
    src="https://cdn.nia.io/widget/nia-widget.iife.js"
    data-tenant="mi-empresa"
    data-api-url="https://api.nia.io"
    data-tenant-manager-url="https://api.nia.io"
    data-position="bottom-right"
  ></script>
</head>
⚠️
Atributos data-* al final del body: Vite compila el IIFE de forma que document.currentScript es null al 97% del bundle. Usa siempre window.NIA_CONFIG si insertas el script antes del </body>.

Prioridad de configuración

Cuando coexisten ambas formas, el orden de prioridad es:

window.NIA_CONFIG → currentScript.dataset.* → querySelector("script[data-tenant]") → defaults hardcoded

Cómo funciona

1. Browser descarga nia-widget.iife.js
↓
2. IIFE lee config desde window.NIA_CONFIG o data-* attrs
↓
3. Crea <div id="nia-widget-host"> con Shadow DOM
↓
4. GET {tenantManagerUrl}/tenants/{tenantId}/widget-config
→ aplica primaryColor, logo, textos al widget
↓
5. Renderiza botón launcher (💬) en esquina configurada
↓
6. Usuario hace clic → abre panel de chat
↓
7. Mensaje del usuario → POST {apiUrl}/v1/chat
→ respuesta del asistente en pantalla

Métodos de configuración

El widget soporta dos métodos de configuración. Internamente, el código en embed.tsx los resuelve en este orden:

// embed.tsx — lógica de resolución de config
const cfg = (window as any).NIA_CONFIG ?? {};
const script = _scriptEl;  // currentScript capturado al inicio del IIFE

const tenantId =
  cfg.tenant ??
  script?.dataset.tenant ??
  "";

const apiUrl =
  cfg.apiUrl ??
  script?.dataset.apiUrl ??
  "http://localhost:8001";

const tenantManagerUrl =
  cfg.tenantManagerUrl ??
  script?.dataset.tenantManagerUrl ??
  "http://localhost:8003";

const position =
  cfg.position ??
  script?.dataset.position ??
  "bottom-right";

Referencia de configuración

tenant string Requerido
ID único del tenant. Se usa para obtener el branding y como contexto en todas las llamadas al backend.
Ejemplo: "riviera-maya-tours"
apiUrl string (URL) Requerido
URL base del orchestrator/API. El widget hace POST {apiUrl}/v1/chat para enviar mensajes.
Default: "http://localhost:8001"
tenantManagerUrl string (URL) Opcional
URL base del tenant-manager. El widget hace GET {tenantManagerUrl}/tenants/{tenant}/widget-config para obtener el branding. Puede ser el mismo servidor que apiUrl.
Default: "http://localhost:8003"
position "bottom-right" | "bottom-left" Opcional
Posición del botón launcher y del panel de chat en la pantalla.
Default: "bottom-right"

Branding por tenant

El widget hace GET /tenants/{tenantId}/widget-config al montar y aplica los valores recibidos como CSS custom properties en el Shadow DOM. Si el endpoint falla, usa los valores por defecto (azul #2563eb).

Respuesta esperada

// GET /tenants/{tenant_id}/widget-config
// → Objeto PLANO (sin wrapper "data")
{
  "primary_color":   "#10b981",        // color del launcher + header
  "logo_url":        null,             // URL imagen o null → emoji 🤖
  "welcome_message": "¡Hola! Soy NIA…", // primer mensaje del asistente
  "placeholder":     "Escribe aquí…",  // placeholder del textarea
  "widget_token":    "tok_abc123"      // Bearer token para /v1/chat
}
🚨
No envolver en {"data": {...}}. El widget lee g.primary_color directamente. Si devuelves {"data": {"primary_color": ...}}, el branding se ignorará y se usarán los defaults.

Cómo se aplica el theming

// Widget.tsx — aplica primaryColor como CSS custom property
const style = { "--nia-primary": branding.primaryColor };

// En widget.css (Shadow DOM)
.nia-launcher {
  background-color: var(--nia-primary, #2563eb);
}
.nia-header {
  background: var(--nia-primary, #2563eb);
}

API — GET widget-config

GET /tenants/{tenant_id}/widget-config

Llamado automáticamente por el widget al montar. No requiere autenticación.

CampoTipoDescripción

widget_tokenstringBearer token para autenticar llamadas a /v1/chat

API — POST /v1/chat

POST {apiUrl}/v1/chat

Endpoint principal al que el widget envía cada mensaje del usuario.

Request

// Headers
Content-Type: application/json
Authorization: Bearer {widget_token}

// Body
{
  "message": "¿Qué tours tienen en Tulum?"
}
ℹ️
El widget NO envía session_id en el body — la sesión se gestiona server-side. El token sirve como identificador de sesión implícito en producción.

Response

// El widget lee response.data (envolver en "data" es obligatorio aquí)
{
  "data": {
    "session_id":        "s_abc123",
    "response":          "Tenemos tours en Tulum desde $89…",
    "fsm_state":         "recommendation",
    "show_lead_form":     false,
    "recommendations":   null,
    "handoff_triggered": false,
    "checkout_url":      null
  }
}
CampoTipoDescripción
responsestringTexto que se muestra como burbuja del asistente
fsm_statestringEstado del FSM: greeting, chatting, recommendation, checkout, handoff
show_lead_formbooleanSi true, el widget muestra el formulario de captura de lead
recommendationsarray | nullLista de productos/tours recomendados (ver tipo RecommendationItem)
handoff_triggeredbooleanSi true, la conversación fue transferida a un agente humano
checkout_urlstring | nullURL de checkout a mostrar si hay producto seleccionado

Componentes internos

El widget está compuesto por los siguientes componentes Preact en src/components/:

Hook useChat

El hook principal que gestiona el estado de la conversación. Está en src/hooks/useChat.ts.

interface UseChatOptions {
  apiUrl:    string;
  token:     string;   // widget_token del tenant
  sessionId: string;   // generado por getOrCreateSessionId()
}

const {
  messages,         // ChatMessage[]  — historial completo
  loading,          // boolean        — esperando respuesta
  fsmState,         // string         — estado actual del FSM
  showLeadForm,     // boolean        — mostrar formulario de lead
  handoffActive,    // boolean        — chat transferido a agente
  checkoutUrl,      // string | null  — URL de checkout activa
  error,            // string | null  — último error
  send,             // (text: string) => Promise<void>
} = useChat({ apiUrl, token, sessionId });

Tipo ChatMessage

interface ChatMessage {
  id:               string;
  role:             "user" | "assistant";
  content:          string;
  timestamp:        number;            // Date.now()
  recommendations?: RecommendationItem[];
  isStreaming?:     boolean;
}

Gestión de sesión

El widget genera un session_id único por pestaña del browser y lo almacena en sessionStorage. Se regenera automáticamente en cada nueva pestaña. 

// Widget.tsx
function getOrCreateSessionId(): string {
  const key = "nia_session_id";
  let id = sessionStorage.getItem(key);
  if (!id) {
    id = `s_${Math.random().toString(36).slice(2)}_${Date.now()}`;
    sessionStorage.setItem(key, id);
  }
  return id;
}
ℹ️
El session_id se pasa al hook useChat pero el widget actual no lo incluye en el body del POST /v1/chat. En producción, el backend lo deriva del JWT del widget_token.

Shadow DOM & aislamiento

El widget crea un Shadow DOM en modo open para aislar completamente sus estilos:

// embed.tsx
const host = document.createElement("div");
host.id = "nia-widget-host";
host.style.cssText = "position:fixed;z-index:2147483647;pointer-events:none;inset:0;";
document.body.appendChild(host);

const shadow = host.attachShadow({ mode: "open" });  // ← Shadow DOM

// Los estilos del sitio host NO afectan al widget
// Los estilos del widget NO afectan al sitio host
ElementoDónde viveDescripción
#nia-widget-hostDOM principalDiv contenedor, position:fixed, cubre toda la pantalla con pointer-events:none
Shadow RootDentro de #nia-widget-hostRaíz del Shadow DOM, contiene estilos + componentes
#nia-rootShadow DOMMount point de Preact
.nia-launcherShadow DOMBotón FAB circular, pointer-events:all
.nia-panelShadow DOMVentana de chat, sólo visible cuando open=true

Setup local

# Desde la raíz del monorepo
cd packages/widget
npm install

# Dev server (hot-reload en puerto 3000)
npm run dev

# Type-check sin build
npm run typecheck
💡
En dev con npm run dev, Vite sirve el widget con HMR. Para probar el IIFE compilado (el que va a producción), usa siempre npm run build.

Build

npm run build
# → dist/nia-widget.iife.js  (~27 KB · gzip: ~10 KB)

La configuración de Vite está en vite.config.ts:

// vite.config.ts
build: {
  lib: {
    entry: "src/embed.tsx",
    name:  "NiaWidget",
    formats: ["iife"],          // bundle autoejectable
    fileName: "nia-widget",
  },
  rollupOptions: {
    output: { inlineDynamicImports: true },
  },
}
⚠️
El formato iife de Vite coloca el código del módulo entry (embed.tsx) al final del bundle. Por eso document.currentScript ya es null cuando se ejecuta. Usa siempre window.NIA_CONFIG como mecanismo de configuración preferido.

Demo server

El proyecto incluye un servidor de demo en la raíz: demo_server.py. Sirve un sitio de turismo falso ("Riviera Maya Tours") con el widget integrado y hace de proxy hacia el model-adapter. 

# 1. Iniciar model-adapter (LM Studio en puerto 1234)
MODEL_PROVIDER=lmstudio \
PYTHONPATH=services/model-adapter:. \
uvicorn app.main:app --app-dir services/model-adapter --port 8005

# 2. Iniciar demo server
python demo_server.py

# 3. Abrir en browser
open http://localhost:8080
EndpointDescripción
GET /Sirve demo.html (sitio de turismo)
GET /packages/widget/dist/nia-widget.iife.jsBundle del widget compilado
GET /tenants/{id}/widget-configConfig de branding hardcodeada para la demo
POST /v1/chatProxy a model-adapter:8005, mantiene historial en memoria
POST /v1/sessions/{id}/leadNo-op (imprime en consola)
ℹ️
El demo server usa window.NIA_CONFIG (no data-* attrs) para configurar el widget, lo que garantiza que funcione correctamente con el bundle IIFE de Vite.
